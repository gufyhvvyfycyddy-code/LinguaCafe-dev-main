# LinguaCafe Dev Main 开发工作流

## 仓库定位

- 本仓库是**未来主开发仓库**，用于 LinguaCafe FSRS / sense-only 改造项目的全部后续开发。
- 旧仓库（`LinguaCafe-local`）是**历史仓库**，不再自动推送。
- 后续开发完成一轮并通过网页端 GPT 验收后，应优先 push 到本仓库。
- 只有用户明确要求时才更新。

## 每次任务前

1. **网页端 GPT 必须检查 GitHub 最新代码。**
   - 不要凭聊天记录判断代码状态。代码状态以 GitHub 最新 commit 为准。
   - 如果本地有未 push 的改动，先确认是否已完成验收。

2. **推荐模型和档位。**
   - 根据任务类型从模型档位表中选用合适的模型。
   - 在拆解任务时一并注明推荐模型。

3. **提出产品问题。**
   - OpenCode 提示词之外，必须面向用户提出产品层面的问题。
   - 用户是产品设计者，不需要判断代码细节。
   - 技术说明要面向功能、体验、风险、取舍。

## 每轮执行后报告格式

完成一轮开发任务后，必须报告以下内容：

1. 当前 commit
2. 是否 push
3. `git status`
4. 修改了哪些文件
5. 实现内容摘要
6. 测试结果（phpunit 输出摘要）
7. 浏览器 smoke 结果（手动验证的关键页面）
8. 合规确认
9. 是否进入下一任务：**否**

## 用户确认流程

1. 用户确认 Accept 后，下一轮继续。
2. 未经验收不得 push。
3. 用户不需要判断代码。用户的作用是确认产品方向、手动体验、敏感配置输入。

## 禁止事项

- 不要凭聊天记录判断代码状态 —— 以 `git status` 和 GitHub 最新代码为准。
- 不要凭预期行为判断完成 —— 必须通过测试和浏览器验证。
- 不要自动 DCP。
- 不要运行 notification script。
- 不要 `migrate:fresh`、`db:wipe`、清库。
- 不要默认 push 旧仓库。

## OpenCode Slim 多 Agent 使用原则

1. **Plan**：只读侦察、列计划、拆任务。
2. **Build**：在明确边界内执行修改。
3. **Composer / Orchestrator**：适合多阶段复杂任务。
4. 多 Agent 可以并行侦察和审查，但写文件时必须明确文件边界，禁止 Agent 间互相覆盖或擅自扩大范围。
5. 使用前建议刷新模型列表。
6. 进入 OpenCode 后建议先执行只读 smoke。

## 公开仓库与数据库边界

本仓库可以公开，但 AI 和本地 Agent 必须区分三类内容：

1. **可公开**：源码、文档、安全脚本、schema-only SQL、匿名示例数据。
2. **默认不公开**：本机私有配置、账号凭据、OpenCode 私有配置。
3. **默认不公开**：真实学习数据库、storage 个人学习数据、完整迁移包。

如果用户说"数据库可以公开"，必须追问或确认具体含义：是公开结构，还是公开真实数据。默认按公开结构处理。
